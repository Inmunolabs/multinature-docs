# Minuta de reunión

**Tema principal:** Multinature — flujos de registro, uso de plataforma, contenido comercial y materiales para especialistas  
**Tipo de documento:** Minuta de trabajo  
**Observación:** La transcripción original tiene ruido y fragmentos incompletos, así que algunos puntos fueron consolidados por contexto.

## 1. Objetivo de la reunión
Alinear decisiones sobre:
- registro de usuarios y pacientes en Multinature,
- limitaciones actuales del flujo con teléfono vs correo,
- funcionalidades clave para especialistas y usuarios,
- estrategia de contenido y grabaciones,
- materiales comerciales y educativos para impulsar uso de la plataforma y venta de productos.

## 2. Temas tratados

### 2.1 Registro de usuarios con teléfono vs correo
Se discutió la necesidad de permitir registros con número telefónico, principalmente para personas que solo quieren comprar productos y no necesariamente usar toda la plataforma.

#### Conclusiones
- **Hoy el sistema depende mucho del correo electrónico**, porque gran parte de las notificaciones y validaciones se hacen por email.
- Para pacientes dados de alta por especialistas, **actualmente el flujo sigue siendo por correo**:
  - el especialista da de alta al paciente,
  - al paciente le llega un código por correo,
  - crea contraseña y activa su cuenta.
- **Todavía no existe el flujo equivalente por SMS**.
- Se mencionó que **sí o sí se quiere resolver el tema del teléfono**, pero no necesariamente rehaciendo de inmediato todo el sistema.

#### Enfoque acordado
- Para personas que **solo van a comprar** y no tienen correo, se considera mejor:
  - que el **especialista pueda generarles el pedido como invitado**, o
  - crear un flujo más simple de compra donde se pida **teléfono o correo**.
- Para pacientes que van a usar la plataforma como tal, el correo sigue siendo necesario por ahora.

### 2.2 Funcionalidades clave para especialistas
Se habló de reforzar la plataforma como herramienta integral para especialistas.

#### Funcionalidades mencionadas
- alta de pacientes,
- agenda / citas,
- consultas,
- seguimiento del plan,
- historial clínico,
- simulación de consulta,
- recomendaciones,
- menús / platillos,
- sugerencia de productos,
- clínica / consultorio,
- datos bancarios y fiscales,
- plan de compensación,
- vista de red,
- compartir enlaces para invitar especialistas o usuarios.

#### Enfoque
La comunicación inmediata debe priorizar que **Multinature es una plataforma para especialistas**, no solo una tienda.

### 2.3 Funcionalidades clave para usuarios / pacientes
Se habló de qué puede hacer un usuario general dentro del sistema.

#### Funcionalidades mencionadas
- seguimiento a su plan,
- calendario de actividades,
- citas,
- especialistas asignados,
- historial y seguimiento,
- reemplazos de alimentos,
- compra de productos,
- compras recurrentes,
- red / enlace compartido,
- datos personales, bancarios y fiscales,
- integraciones,
- soporte.

#### Observaciones
- El módulo de reemplazos se considera importante, especialmente en nutrición.
- La experiencia de usuario debe ser más clara para acciones comerciales como compartir enlace y comprar.

### 2.4 Modelo comercial y red de comisiones
Se revisó cómo comunicar el sistema de compensación y el comportamiento esperado de especialistas y usuarios.

#### Ideas centrales
- El especialista debe:
  - meter a sus pacientes a la plataforma,
  - usarla para gestionarlos,
  - venderles productos,
  - beneficiarse del esquema de comisión.
- También existe la posibilidad de que usuarios generales compartan su liga y generen compras.
- Se habló de que hay dos comunicaciones distintas:
  1. **especialistas**: usar la plataforma + vender + gestionar pacientes,
  2. **usuarios generales**: comprar, recomendar productos y ganar comisiones.

#### Alineación
- La **prioridad actual de comunicación** será hacia especialistas.
- La comunicación para usuarios generales / afiliación por interés económico **puede venir después**.

### 2.5 Contenido, marketing y grabaciones
Se incorporó a Minerva / Paloma al contexto de creación de contenido para Multinature.

#### Se definió que habrá dos líneas de contenido
1. **Contenido tutorial / masterclass**
   - enfocado en explicar cómo usar la plataforma,
   - grabado tipo tutorial con pantalla + explicación,
   - dividido en cápsulas cortas.

2. **Contenido comercial / viral**
   - enfocado en productos,
   - beneficios,
   - dolores del usuario,
   - plan de compensación,
   - mensajes para atraer especialistas y potenciales recomendadores.

#### Lineamientos comentados
- El lenguaje debe ser **coloquial, claro y con gancho**.
- Videos cortos ideales de **40 a 45 segundos** para redes.
- La masterclass puede dividirse en piezas más pequeñas.
- Se sugirió que el contenido corto lleve tráfico a una pieza más completa.

### 2.6 Landing pages y materiales de apoyo
Se habló de que probablemente hará falta una mejor estructura de páginas de aterrizaje.

#### Propuesta
- crear una landing para **especialistas**,
- crear otra para **usuarios / público general**,
- incluir beneficios,
- CTA claro,
- posible video incrustado,
- conexión con campañas.

#### Razonamiento
La página actual no parece suficiente para campañas de conversión o explicación rápida del sistema.

### 2.7 Problemas detectados en pruebas
Se mencionaron varios problemas operativos / de UX.

#### Problemas comentados
- usuarios que no pudieron completar registro por problemas con código de verificación,
- dificultad para regenerar o reutilizar código,
- problemas al abrir links compartidos en ciertos navegadores,
- sospecha de fallas específicas en Safari,
- enlaces que desde algunos contextos mandaban a not found,
- poca visibilidad del botón o acceso para compartir liga,
- necesidad de que el link de venta sea más directo y fácil de usar.

#### Recomendaciones surgidas en la conversación
- revisar el flujo de verificación por correo,
- validar expiración / reenvío de código,
- probar enlaces en Safari,
- hacer más visible el CTA de compartir enlace,
- mejorar UX de “recomienda y gana”.

## 3. Acuerdos

### Acuerdos funcionales
- Mantener por ahora el flujo principal de pacientes con **correo electrónico**.
- Explorar / agregar opción para que el **especialista pueda crear pedidos para invitados**.
- Resolver el caso de usuarios que solo quieren comprar y no usar toda la plataforma.
- Mantener la separación entre perfiles de:
  - **especialista**
  - **consumidor / usuario general**

### Acuerdos de producto / negocio
- La prioridad de comunicación será **especialistas**.
- El mensaje principal debe ser:
  - usa la plataforma para gestionar pacientes,
  - recomienda productos,
  - genera ingresos adicionales.
- La comunicación para usuarios generales que compran o recomiendan sin ser especialistas queda como línea secundaria.

### Acuerdos de contenido
- Crear contenido tutorial de uso de plataforma.
- Crear contenido comercial / viral con enfoque en:
  - problema de salud,
  - beneficio del producto,
  - plan de compensación,
  - oportunidad de ingreso.
- Dividir contenido largo en cápsulas más cortas.

## 5. Riesgos detectados
- Querer meter registro por teléfono “a medias” puede romper flujos que hoy dependen de correo.
- La experiencia actual de registro / compartir links todavía tiene fricción.
- Si la comunicación mezcla demasiado especialista + consumidor final, el mensaje comercial se diluye.
- El contenido puede salir mal si primero no se aterriza bien:
  - qué se quiere vender,
  - a quién,
  - con qué CTA,
  - y qué flujo real sí funciona hoy en producto.
